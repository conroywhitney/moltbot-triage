## Description

## Description

Cron jobs created by the agent via the `cron` tool's `add` action are silently created in a disabled state when the LLM omits the `enabled` field.

The tool description tells the LLM:
```
"enabled": true | false   // Optional, default true
```

But the code in `normalize.js` → `normalizeCronJobCreate()` never applies this default. The `applyDefaults` block (which sets defaults for `wakeMode` and `sessionTarget`) does not set `enabled = true` when it's missing.

## Root cause

In `cron/normalize.js`, the `enabled` field is only set if explicitly present in the input (line ~77-91). The `applyDefaults` block (line ~98-111) defaults `wakeMode` and `sessionTarget` but not `enabled`.

In `cron/service/jobs.js`, `computeJobNextRunAtMs()` checks `if (!job.enabled)` — since `undefined` is falsy in JavaScript, the job is treated as disabled and `nextRunAtMs` is set to `undefined`. The job never fires.

## Steps to reproduce

1. Enable the `cron` tool for a sandboxed agent
2. Ask the agent to set a reminder (e.g., "remind me in 5 minutes to take out the trash")
3. The agent calls `cron` with `action: "add"` and a job object that omits `enabled`
4. Run `openclaw cron list --all` → job shows as `disabled` with `Next: -`
5. The reminder never fires

## Expected behavior

Jobs created via `cron.add` should default to `enabled: true` when the field is not specified, matching the tool description.

## Suggested fix

In `normalizeCronJobCreate` (or the `applyDefaults` block in `normalizeCronJobInput`), add:

```js
if (options.applyDefaults) {
    if (next.enabled === undefined) {
        next.enabled = true;
    }
    // ... existing defaults for wakeMode, sessionTarget
}
```

## Environment

- OpenClaw v2026.1.30
- Model: `openrouter/google/gemini-2.5-flash-lite` (but model-independent — any LLM that omits `enabled` triggers this)
- Sandbox: `mode: "all"`, `scope: "agent"`
