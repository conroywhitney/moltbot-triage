## Description

## Summary

`openclaw status` reports `Memory │ enabled (plugin memory-lancedb) · unavailable` even though the memory-lancedb plugin is fully functional (memory_recall/memory_store work fine).

## Root Cause

In `src/commands/status.scan.ts`, the memory status scanner only knows how to probe `memory-core`:

```typescript
if (memoryPlugin.slot !== "memory-core") {
    return null;  // ← skips probing for any non-core memory plugin
}
```

When `plugins.slots.memory` is set to `"memory-lancedb"` (or any other plugin), the scanner returns `null`, and `status.command.ts` renders this as "unavailable":

```typescript
if (!memory) {
    const slot = memoryPlugin.slot ? `plugin ${memoryPlugin.slot}` : "plugin";
    return muted(`enabled (${slot}) · unavailable`);
}
```

## Expected Behavior

When a non-core memory plugin is active and working, `openclaw status` should query the running gateway for the plugin's memory status (via RPC) and display accurate information (file count, chunk count, vector/fts state, etc.).

## Reproduction

1. Set `plugins.slots.memory` to `"memory-lancedb"` in `openclaw.json`
2. Run `openclaw status`
3. Memory line shows "unavailable" despite the plugin working correctly

## Environment

- OpenClaw: running from source (main branch)
- OS: Ubuntu 24.04 (Linux 6.8.0-90-generic)
- Node: v24.13.0
