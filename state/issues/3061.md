## Description

## Summary

Add support for passing arbitrary vendor-specific parameters through to the chat completions API body. This would enable features like Venice's `enable_web_scraping` without requiring code changes for each vendor.

## Use Case

Venice.ai supports a `venice_parameters` field in chat completion requests that enables features like automatic web scraping:
```json
{
  "model": "claude-opus-45",
  "messages": [...],
  "venice_parameters": {
    "enable_web_scraping": true
  }
}
```

When enabled, Venice automatically scrapes URLs (web articles, YouTube videos, etc.) included in messages - perfect for link summarization use cases.

## Current Behavior

The config schema supports `agents.defaults.models.<provider/model>.params`, but only whitelisted params are passed through:
- `temperature`
- `maxTokens`
- `cacheControlTtl`

Arbitrary vendor params like `venice_parameters` are ignored.

## Proposed Solution
Add a `providerParams` (or similar) field that gets spread directly into the API request body:
```json
{
  "agents": {
    "defaults": {
      "models": {
        "venice/claude-opus-45": {
          "alias": "Claude Opus 4.5",
          "providerParams": {
            "venice_parameters": {
              "enable_web_scraping": true
            }
          }
        }
      }
    }
  }
}
```
This would require changes in:
1. `extra-params.js` - pass through `providerParams` 
2. pi-ai's `buildParams()` - spread `providerParams` into the request body

## Alternatives Considered

- **Per-vendor support**: Adding explicit `veniceParameters` config - doesn't scale
- **Proxy workaround**: Injecting params via nginx/caddy - adds infrastructure complexity

## Environment
- Clawdbot version: 2026.1.24-3
- Provider: Venice.ai (OpenAI-compatible endpoint)
