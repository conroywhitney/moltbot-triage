## Description

## Summary
- Fixes webchat message persistence when using CLI backends (claude-cli, codex-cli)
- Messages now persist correctly to transcript files and survive page reloads

## Problem
When using CLI backends, assistant messages appeared during the chat session but disappeared after page reload. This happened because:
1. The `chatLink` was not being registered in `onAgentRunStart`, causing agent events to not be properly routed
2. The `agentRunStarted` flag bypassed the normal message persistence path in the gateway

## Solution
1. Register `chatLink` in `onAgentRunStart` callback so agent events are properly mapped to client runIds
2. Add `saveAssistantToTranscript()` function that persists assistant messages directly to the session transcript JSONL file
3. Call this function in `emitChatFinal` when a CLI backend run completes successfully

## Testing
- Tested with claude-cli backend on local OpenClaw instance
- Verified messages persist correctly in session transcript files
- Confirmed messages survive page reload in webchat UI

### @JorgeAlan (2026-02-01)

Thanks for the detailed review! You've identified valid concerns. Here's my analysis:

## Addressing the feedback

### [P0] Wrong session file path
**Valid concern.** The current implementation uses `loadSessionEntry(sessionKey)` which may resolve to a different sessionId than the agent's actual transcript file.

**Proposed fix:** Pass the correct `sessionId`, `storePath`, and `sessionFile` from `chat.ts` through the `ChatRunEntry` when registering the chatLink in `onAgentRunStart`. This ensures we use the exact same path resolution as `appendAssistantTranscriptMessage()`.

### [P2] Transcript file not created if missing
**Valid concern.** First message scenario would silently fail.

**Proposed fix:** Create the transcript file with a proper session header when missing (similar to `ensureTranscriptFile()`).

### [P2] Duplicate messages for non-CLI runs
**Valid concern.** Embedded backends already persist their assistant messages.

**Proposed fix:** The `chatLink` in the registry is only created for CLI runs (via `onAgentRunStart` which is only called by `runCliAgent`). We can use `chatLink` existence as a guard - only persist when `chatLink` is present.

## Implementation plan

1. Extend `ChatRunEntry` type to include `sessionId`, `storePath`, `sessionFile`
2. Pass this info when calling `context.addChatRun()` in `chat.ts`
3. Update `saveAssistantToTranscript()` to:
   - Accept `ChatRunEntry` instead of `sessionKey`
   - Return early if no chatLink (non-CLI run guard)
   - Create transcript file if missing
   - Use the passed session info for correct path

I'll push an updated commit with these fixes. Let me know if you'd prefer a different approach.
