## Description

## Summary

- Adds fast-path for `cron.list` and `cron.status` operations when the store is already loaded
- Prevents RPC timeouts during long-running cron job executions

## Problem

When a cron job executes with `wakeMode: "now"`, it can block for up to 2 minutes waiting for the heartbeat. During this time, all other cron operations queue behind the global lock in `locked.js`, causing Gateway RPC timeouts for simple read-only operations like `cron.list` and `cron.status`.

## Solution

For read-only operations (`list` and `status`), check if `state.store` is already loaded. If so, return the data immediately without acquiring the lock. This is safe because:

1. These operations only read data, they don't modify state
2. The store is only loaded once and kept in memory  
3. Slight staleness (reading during a concurrent write) is acceptable for status/list operations

## Regarding Greptile's feedback

**P0 - `cronEnabled` handling:**

The `status()` fast-path already handles this correctly:
```typescript
nextWakeAtMs: state.deps.cronEnabled ? (nextWakeAtMs(state) ?? null) : null,
```

For `list()`, the fast-path returns jobs regardless of `cronEnabled` - but this matches the existing locked-path behavior exactly. The `warnIfDisabled()` check is only used for mutating operations (`add`, `update`, `remove`, `run`), not for read-only operations. So this is not a regression.

**P2 - Sort order for missing `nextRunAtMs`:**

The `?? 0` fallback in the sort comparator is copied directly from the existing locked-path code to maintain consistency:
```typescript
// Line 87 - existing code
return jobs.toSorted((a, b) => (a.state.nextRunAtMs ?? 0) - (b.state.nextRunAtMs ?? 0));
```

If this should be changed to `Number.POSITIVE_INFINITY`, that would be a separate improvement affecting both paths.
