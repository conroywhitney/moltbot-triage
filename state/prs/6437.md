<h2>Greptile Overview</h2>

<h3>Greptile Summary</h3>

Adds a new `ProcessLimiter` utility (`src/agents/process-limiter.ts`) intended to mitigate unbounded process spawning by enforcing a max concurrency limit, queueing additional spawn requests, and tracking basic health metrics.

The limiter currently functions as a concurrency gate + queue, but the advertised exponential backoff and “health monitoring” aspects are not effectively implemented: backoff is driven by a parameter that isn’t updated internally, and the `healthCheckIntervalMs` config is unused.

<h3>Confidence Score: 2/5</h3>

- This PR is not yet safe to merge because the backoff logic is effectively inactive and the queuing strategy can leak memory under hangs.
- Core limiter behavior (concurrency cap) is present, but the key safety claim (exponential backoff) depends on `currentFailureCount`, which is never incremented internally, so it won’t engage in typical usage. Additionally, the queue holds on to pending Promises indefinitely if capacity never frees, which can worsen the resource-exhaustion scenarios this aims to prevent. These are fixable but affect correctness and resilience.
